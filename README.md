# 限流算法完整实战演示项目

## 项目概述

这是一个纯Java实现的限流算法演示项目，不依赖任何外部框架或中间件。项目结构极简，适合学习限流原理、调试和自测。

## 四层限流核心能力

### 1. 单IP限流
- **原理**: 基于滑动窗口统计每个IP每秒请求数
- **作用**: 防止单个IP恶意高频访问

### 2. 单账号限流
- **原理**: 基于滑动窗口统计每个账号每秒请求数
- **作用**: 防止单个用户刷屏或恶意操作

### 3. 全局QPS封顶限流
- **原理**: 统计系统整体每秒请求数
- **作用**: 保护系统整体承载能力

### 4. 令牌桶算法限流
- **原理**: 以固定速率生成令牌，请求获取令牌后通过
- **特点**: 允许一定程度的突发流量，但有上限
- **优势**: 适合业务突发场景，如秒杀活动

### 5. 漏桶算法限流
- **原理**: 请求进入桶中，以固定速率流出
- **特点**: 平滑流量输出，削峰填谷
- **优势**: 防止突发流量击垮下游系统

## 校验顺序设计

请求按以下顺序依次经过各层限流：

1. **单IP/单账号限流** → 2. **全局QPS封顶** → 3. **令牌桶限流** → 4. **漏桶限流**

### 设计原因
- **优先级**: 用户级限制优先于系统级，防止恶意用户影响整体服务
- **粒度递增**: 从细粒度到粗粒度，层层防护
- **兜底机制**: 漏桶作为最后防线，确保流量平滑输出

## 令牌桶 vs 漏桶

| 特性 | 令牌桶 | 漏桶 |
|------|--------|------|
| 流量特点 | 允许突发，但有上限 | 强制平滑输出 |
| 适用场景 | 业务突发允许 | 严格控制输出速率 |
| 积攒能力 | 可积攒令牌 | 不积攒，超限丢弃 |
| 实现复杂度 | 中等 | 中等 |

### 令牌桶最大容量限制
- **问题**: 如果不限制令牌桶容量，长时间低流量后会积攒大量令牌，导致突发大流量击穿系统
- **解决方案**: 设置最大容量，限制积攒令牌上限
- **效果**: 平衡突发处理能力和系统保护

## 多层组合防护设计

### 设计逻辑
1. **分层防护**: 不同维度限制，全面覆盖攻击面
2. **渐进式拦截**: 优先拦截明显恶意请求，减少后续处理压力
3. **弹性缓冲**: 令牌桶提供弹性，漏桶保证兜底
4. **雪崩防护**: 多层限制防止单点故障导致系统雪崩

### 实际效果
- **正常流量**: 顺利通过所有层级
- **合理突发**: 令牌桶吸收，漏桶平滑输出
- **恶意攻击**: 前几层快速拦截，保护核心系统
- **极端情况**: 漏桶兜底，确保系统稳定

```bash

javac -d . src/main/java/com/example/ratelimiter/*.java

# 运行
java com.example.ratelimiter.Simulation
```

## 配置说明

所有参数在 `Config.java` 中统一配置：

- **开关控制**: 每层限流可独立开启/关闭
- **阈值设置**: 灵活调整各种限制参数
- **实时生效**: 修改后重启程序生效

## 测试场景

项目内置5种典型场景模拟：

1. **平稳正常流量**: 验证基础限流功能
2. **业务合理突发**: 测试令牌桶弹性
3. **恶意单IP攻击**: 验证IP级防护
4. **账号刷屏请求**: 测试用户级限制
5. **令牌桶攒满突发**: 验证容量限制效果

## 日志输出

每条请求输出格式：
```
请求放行 - IP: 192.168.1.1, 账号: user123
请求拦截 - IP: 192.168.1.1, 账号: user123 | 被拦截层: 单IP限流 | 规则: 单IP每秒最大请求数: 10
```

## 扩展指南

- **新增限流策略**: 实现 `RateLimiter` 接口，添加到 `RateLimitManager`
- **自定义参数**: 在 `Config` 中添加配置项
- **调整顺序**: 修改 `RateLimitManager.check()` 方法
- **性能优化**: 可引入缓存或分布式存储

## 注意事项

- 生产环境建议使用更精确的定时器
- 高并发场景可考虑分片或分布式限流
- 内存使用：各限流器维护独立计数器，注意内存占用
